# Standup board rendering — a full multi-plan table, ephemeral by default and persistable on demand

> Status: **done** · created 2026-07-07 · completed 2026-07-07
> Model: fable · Effort: medium
> Home: cross-repo system (skills are global in `~/.claude/skills/`), so the plan lives in the
> dotfiles repo. Every slice edits `~/.claude` only.
> Why: juggling several plans, the on-demand-only render isn't enough — the user is already
> hand-building a persistent board (`docs/plans/00-master-index.md` in workbench). Give the cheap
> glance always and the shareable artifact when asked, without re-introducing drift.
> Reasoning: `docs/notes/standup-board-rendering-reasoning.md` — its decision (option C,
> split-by-cost) and carried risks are inputs here, not up for re-litigation.

## Definition of done

`/standup` (no arg) prints a full multi-plan table of all active plans, not just WIP + the one
NEXT — and the `▶ NEXT` pick is unchanged. `/standup --board` regenerates a persisted
`docs/BOARD.md` from the plan files; a routine `/standup` writes no file. The persisted file
cannot be mis-parsed as a plan and cannot silently drift.

## Acceptance criteria

- [ ] Running `/standup` shows every active plan as a row (plan, status, slices done/total, next
      open slice, last-touched, model/effort) — while still leading with the single `▶ NEXT` line.
- [ ] Running `/standup --board` writes/refreshes `docs/BOARD.md` with the same table and a
      "generated — do not edit" header, regenerated from the plan files each run.
- [ ] A routine `/standup` (no flag) writes no file and dirties no git state.
- [ ] The board file is never parsed as a plan by `/standup`'s own plan sweep.
